How the Vault is built

A one-person, end-to-end analytics platform — from a rate-limited ingest of the NBA's official stats API to an interactive UI. This is the architecture, the data-quality work, and the design tradeoffs, written for an engineer evaluating the project.

Overview

NBA Player Vault ingests every game of a player's career from stats.nba.com, warehouses it, and serves it through player profiles, head-to-head comparisons, leaderboards, and a draft game. It covers all 30 NBA teams plus the WNBA — on the order of 540 players and 163,000+ game-level records.

The point of the project is to demonstrate a full data platform built and shipped by one person: ingestion, warehousing, a shared access layer, a REST API, and a UI — with the real data-quality and reliability problems that come with live sports data.

Architecture

The system is a one-directional, medallion-style pipeline. Each stage has a single responsibility, and nothing downstream reaches around the layer below it.

stats.nba.com (nba_api)
   │   rate-limited, resumable pull
   ▼
nba_daemon.py ──► per-player canonical CSVs        ← RAW landing layer
   ▼
warehouse.py ───► player_vault.duckdb              ← CURATED warehouse
   ▼              (games · players · season_averages view)
data.py  ← single shared query layer (read-only)
   ├──► api.py        (FastAPI — REST + OpenAPI docs)
   └──► vault_app.py  (Streamlit + Plotly UI)

The key decision is the shared access layer (data.py): both the API and the UI import the same query functions, so SQL logic is never duplicated or allowed to drift between frontends. Two small enrichment scripts add the dimensions the box-score endpoints don't carry — position and current team — cached as JSON sidecars and loaded read-only, so they never require a warehouse rebuild.

Data model

The warehouse is intentionally small and denormalized for analytics:

  • games — one row per player-game, canonical schema (points, rebounds, assists, steals, blocks, turnovers, makes/attempts, computed Game Score, opponent, result, season, league).
  • players — one row per player (id, headshot URL, league, first/last season, games played).
  • season_averages — a view aggregating games to per-season lines, with shooting percentages re-derived from totals (see Data quality).

Ingestion & resilience

The ingestion daemon is built to run unattended against a rate-limited, sometimes-flaky API without getting blocked or losing work:

  • Global rate gate — a throttle enforces ~2–3s between calls regardless of which code path triggers them.
  • Retry / backoff / cooldown — transient failures retry with growing delays; a run of consecutive failures triggers a longer cooldown rather than hammering the API.
  • Idempotent + resumable — each player is an independent CSV, so a run dedupes against what's on disk and pulls only what's missing. A load can be paused and resumed with no double-work.
War story · resilience

During a load of ~200 players, my internet physically dropped mid-run. Nothing was lost: the daemon's retry/backoff rode through the gap, an orchestrator-level retry pass re-pulled the single player that failed outright, and the resumable design let the run continue exactly where it left off when the connection returned.

An orchestrator (add_teams.py) chains the whole thing end-to-end for a set of teams — discover roster → dedupe → pull new players → retry failures → rebuild warehouse → enrich positions & teams — in one self-contained, network-only run.

Data quality

Live sports data is messy in specific, instructive ways. Two of the most useful bugs:

Bug · statistical correctness

Shooting percentages were initially computed as the average of per-game percentages — which is wrong, because it ignores attempt volume. One player's career free-throw percentage read 80.8% when the true value was 90.8%.

The fix re-derives every percentage from totals, and now matches Basketball-Reference exactly:

-- shooting % as total makes / total attempts, never an average of per-game %
round(sum(fg)  * 1.0 / nullif(sum(fga),  0), 3) AS fg_pct,
round(sum(fg3) * 1.0 / nullif(sum(fg3a), 0), 3) AS fg3_pct,
round(sum(ft)  * 1.0 / nullif(sum(fta),  0), 3) AS ft_pct
Bug · identity resolution

Player names break on accents (Jokić), hyphens (Gilgeous-Alexander), and rookies missing from the bundled static list. The resolver applies Unicode NFKD accent-folding and hyphen normalization, then falls back from the static list to a live API lookup so brand-new players still resolve.

Validation is anchored to an external authority: aggregates are checked against Basketball-Reference, and the data source itself is the official stats.nba.com API rather than a scraper — a deliberate choice for authority and stability.

Design decisions & tradeoffs

DuckDB over a server database

Embedded, zero-ops, columnar — millisecond analytical scans and native CSV/Parquet reading, ideal for a single-user analytical workload. The tradeoff is that it's single-writer: the warehouse rebuild needs exclusive access, so it can't run while the app holds a read connection. I sequence around that explicitly (and it's exactly the kind of locking constraint worth understanding rather than hiding).

CSV landing layer

Per-player CSVs are human-inspectable, diff-friendly in git, and a natural incremental unit — you can re-pull one player without touching the rest. They're the raw tier that feeds the curated warehouse.

A shared query layer

Putting every query in data.py keeps the API and UI honest — one source of truth, no drift. It also makes the whole thing testable in isolation.

nba_api over scraping

The official endpoints are structured and authoritative; the cost is strict rate limits, which the daemon is built to respect.

Testing

The 82-0 game engine is a pure function with no I/O, so its scoring logic is directly unit-testable. UI flows — navigation, team drill-down, player profiles, edge cases — are exercised headlessly with Streamlit's AppTest. That harness is how a latent crash was caught: a deep-bench rookie with a zero-attempt season produced NULL percentages that broke a pandas type conversion, which is now NaN-safe.

Scale & performance

The expensive operation is API ingestion, which is already incremental — one file per player, dedupe on resume. The dataset itself is small in absolute terms (tens of megabytes at ~540 players), so DuckDB serves queries in milliseconds. The next scaling step is Parquet plus incremental upserts to retire full warehouse rebuilds; a move to Postgres would only be warranted for concurrent multi-user writes.

Running it locally

# serve the app (reads the committed warehouse)
pip install -r requirements.txt
streamlit run vault_app.py

# rebuild / extend the data (pipeline deps)
pip install -r requirements-pipeline.txt
python add_teams.py Celtics 76ers Heat   # discover → pull → warehouse → enrich

Roadmap

  • ML career-trajectory projection on the season data.
  • "Chat with the Vault" — an LLM text-to-SQL layer over the warehouse.
  • Player embeddings & similarity clustering.
  • An all-time-greats tier (the API supports historical players with full box scores).
  • Parquet + incremental upserts; a React frontend on the same REST API.
