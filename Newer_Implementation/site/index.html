The hard parts

Real data. Real bugs. Real fixes.

The difference between a toy project and a platform is what breaks — and whether you caught it. Three problems I hit and solved:

Data quality

A free‑throw % read 80.8% — the truth was 90.8%

Cause: percentages were computed as the average of per‑game percentages, which ignores attempt volume.

Fix: re‑derive every % as total makes ÷ total attempts, across the warehouse view, career, and leaderboards — now matches Basketball‑Reference exactly.

Dirty data

Names break on accents, hyphens, and missing rookies

Cause: "Jokić", "Gilgeous‑Alexander", and players absent from the bundled static list all failed naive matching.

Fix: a resolver using Unicode NFKD accent‑folding, hyphen normalization, and a static‑list → live‑API fallback.

Resilience

My internet dropped mid‑run. Zero data lost.

Cause: a real network outage during a large ingest of hundreds of players.

Fix: daemon retry/backoff rode through it, an orchestrator retry pass re‑pulled the one casualty, and the load is fully pausable/resumable — it dedupes and pulls only what's missing.

The toolbox

Every choice has a tradeoff I can defend

Not a logo parade — the reasoning behind each tool, including where it breaks down.

DuckDB
Embedded, zero‑ops, columnar — millisecond analytical scans, reads CSV/Parquet natively.
Single‑writer; I'd move to Postgres for concurrent multi‑user writes.
nba_api
The official stats.nba.com endpoints — authoritative and structured.
Strict rate limits; handled with a global gate, backoff, and checkpointing.
FastAPI
Typed REST with auto‑generated OpenAPI docs, near‑zero boilerplate.
Read‑only here; the heavy lifting lives in the shared data layer.
Streamlit + Plotly
Ships an interactive, charted data UI fast as a solo developer.
Less layout control than React — a planned future frontend.
